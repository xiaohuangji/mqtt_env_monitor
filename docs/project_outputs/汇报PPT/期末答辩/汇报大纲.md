# 期末答辩汇报大纲

> 题目12《基于 MQTT 的多节点环境监测系统设计》
> 副标题：**面向校园场景的最低成本网络层方案**
> 汇报时间：2026-06-18（周四）｜ 约 18 min + Demo ｜ 幻灯片 `final_slides.tex`（约 32 页，以 .tex 为准）

## 一句话主线

面向多节点环境监测，我们不追求最大最强，而是回答：**在满足采集需求与服务质量（到达率/延迟/重连）的前提下，网络层服务器的最小资源——也就是最低成本——是多少？**
做法：需求分析立问题 → 边设计边用实测佐证 → 用一个校园案例落地测算 → 现场演示真机。

## 叙事顺序（当前版）

**需求分析 → 方案设计与实测验证（每个设计紧跟实验佐证）→ 具体案例 → 现场演示。**

- 开头直接进 **需求分析**（不再单独放"提出问题"hook 页）。
- 成本是**"中期检查老师提出 → 我们及时纳入"**的因果，落在「纳入成本」页（需求分析收尾）。
- **实验不再单独成章**：每讲完一个设计，紧跟一页「实测佐证 · XXX」用真实曲线证明。
- 末尾拆成两部分：**具体案例**（校园 2 万节点测算）和 **现场演示**（ESP32 单节点 Live Demo）。
- **方案设计以「为什么选 MQTT（对比 HTTP）」开场**：用需求对比 HTTP/MQTT，回应题目「比较 MQTT 与 HTTP」要求，自然引出以 MQTT/EMQX 为核心的系统架构。

## 案例与关键数字（用于"具体案例"部分）

| 项 | 取值 |
| --- | --- |
| 案例 | 渭水校区每 100 m² 一节点；面积约 200 万 m² → **约 2 万个节点** |
| 参数 | M=4 类、周期 T=60 s、QoS1、json-min、监控端 S=10 |
| 网络负载 | R≈1333 条/s；**W≈26.4 Mbps**；月流量 ≈ 8.6 TB；磁盘 ≈ 345 GB |
| 服务器最小配置 | **2核4G** + ≥400 GB 盘 + ≥33 Mbps |
| 成本落点 | **一台约 600–1000 元的 2核4G 迷你主机**；瓶颈在带宽/存储，不在 Broker |

> ⚠️ 面积口径（200 万 m²）可按实际可监测面积调整，N 与下游数字联动重算。

## 准确率口径（重要）

- ESP32 实测 **95.0% 到达率**只计运行期（07:19–22:18）的 2 次网络中断，**未掺入 6-16 的服务器欠费停机**。
- 原则：网络短时中断计入（系统靠 QoS1+重连扛住、仍达标）；**欠费停机是外部故障，剔除不计**。
