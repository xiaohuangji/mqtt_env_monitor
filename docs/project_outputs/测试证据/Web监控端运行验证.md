# Web 监控端运行验证

## 1. 编写目的

本文档记录 Web 监控端模块 `src/monitor/web_monitor.py` 的运行验证过程，对应 Issue #18。验证内容包括：

1. 监控端能否订阅 `env_monitor/+/+`，实时显示多个节点、多种数据类型的最新数据。
2. 页面能否按中文名称显示数据类型，并显示节点编号、数据值、单位、上传时间和接收时间。
3. 节点停止上传后，页面能否将对应数据标记为离线/过期。
4. 后端能否记录带接收时间的消息日志，为后续到达率和延迟测试提供数据。

本次验证只覆盖 Web 监控端最小可用版本，不包含历史曲线、数据库存储和正式的 QoS 对比测试。

## 2. 验证环境

| 项目 | 内容 |
| --- | --- |
| 操作系统 | Windows |
| Python | Python 3.11.7 |
| Python 依赖 | `paho-mqtt 2.1.0`、`amqtt 0.11.3`、`flask 3.1.3` |
| Broker | `scripts/mqtt_broker.py`，监听 `127.0.0.1:1883` |
| 数据来源 | `src/simulated_nodes/sim_node.py` 模拟节点 `node01`、`node02` |
| Web 服务 | `http://127.0.0.1:8080/` |
| 验证时间 | 2026-06-10 17:11 ~ 2026-06-10 17:17 |

## 3. 模块说明

Web 监控端由两部分组成：

- `src/monitor/web_monitor.py`：后端程序。使用 paho-mqtt 订阅环境数据主题，维护每个（节点，数据类型）组合的最新数据；使用 Flask 提供监控页面，并通过 SSE（Server-Sent Events）将新数据实时推送到浏览器；每条收到的消息都输出带毫秒级接收时间的日志。
- `src/monitor/templates/index.html`：监控页面。以表格形式显示各节点最新数据，并根据数据新旧程度显示在线状态。

命令行参数：

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `--broker-host` | `127.0.0.1` | MQTT Broker 地址 |
| `--broker-port` | `1883` | MQTT Broker 端口 |
| `--topic` | `env_monitor/+/+` | 订阅主题 |
| `--qos` | `0` | 订阅 QoS 等级 |
| `--web-host` | `127.0.0.1` | Web 服务监听地址 |
| `--web-port` | `8080` | Web 服务监听端口 |
| `--offline-after` | `15.0` | 超过该秒数未收到数据时标记为离线/过期 |
| `--log-file` | 空 | 可选，接收日志追加写入的文件路径 |
| `--client-id` | `web_monitor` | MQTT 客户端标识 |

主要特性：

- 页面表格显示节点编号、数据类型、数据值、单位、上传时间、接收时间、消息序号和状态。
- 内部解析使用英文字段，页面按映射表显示中文数据类型（如 `temperature` 显示为“温度”）；未配置中文名称的类型直接显示英文标识。
- 页面通过 SSE 自动更新，无需手动刷新；连接中断时自动重连。
- 超过 `--offline-after` 秒未收到某节点某类数据时，对应行标记为“离线/过期”，页面顶部统计在线节点数。
- 接收日志精确到毫秒，可与节点发送日志按 `seq` 对照，用于后续消息到达率和平均延迟统计。
- Broker 断开时后端自动重连，重连后自动重新订阅。

## 4. 验证一：双节点实时显示

### 4.1 验证步骤

1. 启动本地 Broker：

   ```powershell
   python scripts\mqtt_broker.py
   ```

2. 启动 Web 监控端（可选 `--log-file` 记录接收日志）：

   ```powershell
   python src\monitor\web_monitor.py --log-file receive.log
   ```

3. 启动两个模拟节点：

   ```powershell
   python src\simulated_nodes\sim_node.py --node-id node01 --interval 3
   python src\simulated_nodes\sim_node.py --node-id node02 --types temperature,light --interval 3
   ```

4. 浏览器访问 `http://127.0.0.1:8080/`。

### 4.2 验证结果

页面顶部状态栏显示：

```text
MQTT Broker：127.0.0.1:1883 ｜ 订阅主题：env_monitor/+/+ ｜ 页面连接：已连接
已接收消息：174 条 ｜ 在线节点：2 个 ｜ 无效消息：0 条
```

数据表格显示 6 行数据（node01 四类 + node02 两类），示例：

| 节点编号 | 数据类型 | 数据值 | 单位 | 上传时间 | 接收时间 | 消息序号 | 状态 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| node01 | 湿度 | 59.5 | %RH | 2026-06-10T17:15:28+08:00 | 2026-06-10T17:15:28.630+08:00 | 298 | 在线 |
| node01 | 光照 | 460.9 | lux | 2026-06-10T17:15:28+08:00 | 2026-06-10T17:15:28.630+08:00 | 299 | 在线 |
| node01 | 噪声 | 45.2 | dB | 2026-06-10T17:15:28+08:00 | 2026-06-10T17:15:28.632+08:00 | 300 | 在线 |

| node02 | 光照 | 454 | lux | 2026-06-10T17:15:29+08:00 | 2026-06-10T17:15:29.166+08:00 | 64 | 在线 |

验证结论：

- 页面能区分显示两个节点、多种数据类型的最新数据，节点每次上传后页面自动更新。
- 数据类型按中文显示，数据值、单位、上传时间、接收时间和消息序号完整。
- 课程要求的“显示节点编号、数据类型、数据值、单位和上传时间”全部满足。

## 5. 验证二：节点离线状态显示

### 5.1 验证步骤

1. 在验证一运行状态下，于 17:15:51 强制结束 `node02` 进程，`node01` 保持运行。
2. 等待超过离线阈值（默认 15 秒）后观察页面。

### 5.2 验证结果

约 18 秒后页面显示：

- `node02` 的光照、温度两行状态变为“离线/过期”（最后数据时间停留在 17:15:50），整行置灰。
- `node01` 四行仍为“在线”，数据持续更新。
- 顶部“在线节点”由 2 个变为 1 个。

验证结论：节点停止上传后，页面能在阈值时间内将对应数据标记为离线/过期，并正确统计在线节点数；其他节点不受影响。

## 6. 接收日志验证

后端每收到一条消息，输出一行带毫秒级接收时间的日志（同时写入 `--log-file` 指定的文件）：

```text
[mqtt] connected to 127.0.0.1:1883, subscribing env_monitor/+/+ qos=0
[recv 2026-06-10T17:11:46.345+08:00] topic=env_monitor/node01/temperature qos=0 payload={"node_id":"node01","data_type":"temperature","value":26.42,"unit":"C","timestamp":"2026-06-10T17:11:46+08:00","seq":1}
```

日志中同时包含消息内的上传时间 `timestamp` 和监控端接收时间，且保留 `seq`，因此可以：

- 按 `seq` 与节点发送日志对照，统计消息到达率；
- 用接收时间减上传时间，统计平均传输延迟。

本次验证运行期间监控端共记录 204 行日志，无效消息 0 条。

## 7. 发现与说明

1. 节点发送侧 `timestamp` 当前为秒级精度，监控端接收时间为毫秒级。后续做平均延迟测试（目标 1 秒以内）时，建议将节点 `timestamp` 提升为毫秒精度，否则延迟统计误差可达 ±1 秒。
2. 监控端对缺字段或非 JSON 的消息会计入“无效消息”并跳过显示，不会影响页面其他数据。
3. 页面离线判断在浏览器端每秒刷新一次，即使没有新消息到达，状态也会自动从“在线”切换为“离线/过期”。

## 8. 小结

本次验证表明 Web 监控端满足 Issue #18 的验收标准：浏览器页面能实时显示各节点的节点编号、数据类型（中文）、数据值、单位、上传时间和接收时间；多节点同时上传时可正确区分并自动更新；节点停止上传后能在阈值时间内提示离线/过期；后端接收日志可与节点发送日志按 `seq` 对照。至此，“模拟节点发布 → Broker 转发 → Web 监控端显示”的完整链路已经打通，可支撑中期检查演示，并为后续 QoS 对比、消息到达率和平均延迟测试提供基础。
