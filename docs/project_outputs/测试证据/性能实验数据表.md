## 2. 协议开销抓包矩阵（每轮 20 条消息，n=20）

口径说明：字节数为**服务器网卡观测的整帧字节**（含以太网头 14B；TCP 无时间戳选项，头 20B）。节点每轮 4 类数据近乎同时发出，TCP 出现真实的粘包合并，下表"边际开销"= (方向总字节 − 连接建立/断开字节) ÷ 20，**已包含粘包摊薄与 ACK 分摊**，可直接用于带宽模型。

| 轮次 | 配置 | 上行边际 B/条 | 下行边际 B/条 | 合计 B/条 |
| --- | --- | --- | --- | --- |
| 1 | v5 + json + Alias，QoS 0 | 167.4 | 35.1 | **202.5** |
| 2 | v5 + json + Alias，QoS 1（主基准） | 196.5 | 65.4 | **261.9** |
| 3 | v5 + json + Alias，QoS 2 | 245.6 | 122.7 | **368.3** |
| 4 | v5 + json-min + Alias，QoS 1 | 137.6 | 65.4 | **203.0** |
| 5 | v5 + msgpack + Alias，QoS 1 | 120.5 | 65.4 | **185.9** |
| 6 | v5 + json 无 Alias，QoS 1 | 217.8 | 65.4 | **283.2** |
| 7 | 3.1.1 + json，QoS 1 | 216.8 | 63.4 | **280.2** |
| 8 | 心跳观测（keepalive 15s） | PINGREQ 60 B + PINGRESP 56 B = **116 B/周期**（理论 112 B + 以太网最小帧填充） | | |

单帧参考值（未粘包时）：PUBLISH(json,Alias,QoS1) 188–189 B；PUBACK v5 60 B（v311 58 B）；QoS2 的 PUBREC/PUBREL/PUBCOMP 各 60–66 B；纯 TCP ACK 54–60 B，实测约 0.7 个/条消息。

### 2.1 对比结论（QoS 1 下）

| 对比 | 数值 | 结论 |
| --- | --- | --- |
| msgpack vs json | 261.9 → 185.9 B | **节省 29%** |
| json-min vs json | 261.9 → 203.0 B | 节省 22% |
| Topic Alias 开 vs 关 | 283.2 → 261.9 B | 每条省 21.3 B（≈8%，主题越长收益越大） |
| QoS 0 / 1 / 2 | 202.5 / 261.9 / 368.3 B | QoS1 比 QoS0 贵 29%，QoS2 比 QoS1 贵 41% |
| MQTT 5.0 vs 3.1.1（同配置无 Alias） | 283.2 vs 280.2 B | 基本持平，5.0 的收益来自 Alias 等特性而非裸协议 |

## 4. amqtt 对照组（MQTT 3.1.1 客户端，`-V 4`）

### 4.1 一个意外发现：amqtt 对 MQTT 5.0 CONNECT 静默挂起

首轮对照实验中 emqtt_bench 默认以 MQTT 5.0 连接 amqtt（仅支持 3.1.1）：amqtt **不拒绝、不应答 CONNACK，TCP 连接静默挂起**——`ss` 能看到数万 ESTABLISHED 连接，但 MQTT 层从未建立，客户端无从得知失败。工程结论：协议版本不匹配时旧方案的失败模式不可观测，这本身就是生产环境的隐患。以下数据为改用 `-V 4`（3.1.1）后的有效重测。

按建模文档场景⑥实景复现：**20 000 个发布连接**（每连接 15 s 一条，68 B payload，QoS 1，聚合 1 333 条/秒）+ **10 个订阅端**全量转发（13 333 条/秒下行），持续 30 分钟，每 5 秒采样。

| 指标 | 实测值 | 结论 |

| 连接稳定性 | 20 010 个连接 **全程零掉线**（357 个采样点 min=max） | 30 min 稳定性指标达标（万级规模下） |

| 内存 | 启动 935 MB → 平台期 **1 475 MB**（最后 20 秒平稳） | 见下方模型修正说明 |

**与计算器推荐配置互验**：计算器对场景⑥输出 2C4G——实测 CPU 占 2 核的 53%、内存占 4 GB 的 36%，**推荐配置成立且留有余量**。

**模型修正点**：带流量时的实测内存（1 475 MB）约为空闲模型裸值（583 MB）的 2.5 倍。本次（2026-06-12）已用 §3.3 满负载爬坡把这条"满负载曲线"完整测出并回填计算器，内存估算从"空闲 × 安全系数"升级为"空闲下界 ~ 满负载上界"双模型，不再依赖单点修正。

## 8. QoS × 丢包率（2026-06-12 加测，netem 注入 loopback 丢包）

用自研 sim_node（发带 `seq` 的消息）+ latency_probe（按 seq 精确统计到达率）。`tc netem` 在 loopback 注入单跳丢包，每格 1 000 条消息（5 节点 × 200 条），sim_node 与 probe 同 QoS。Topic Alias 关闭以隔离 QoS 变量。

| 单跳丢包 | QoS | 到达率 | 平均延迟 ms | P95 延迟 ms |
| --- | --- | --- | --- | --- |
| 0% | 0/1/2 | 100% / 100% / 100% | 0.6 / 0.7 / 1.5 | 1.1 / 1.1 / 1.2 |
| 2% | 0/1/2 | 100% / 100% / 100% | 28.5 / 36.5 / 73.9 | 205.8 / 207.8 / 232.0 |
| 5% | 0/1/2 | 100% / 100% / 100% | 105.9 / 82.1 / 188.9 | 418.9 / 343.9 / 473.8 |
| 10% | 0/1/2 | 100% / 100% / 100% | 365.1 / 310.2 / 462.3 | 1 098.9 / 895.9 / 1 102.1 |
| 15% | 0/1/2 | 100% / 100% / 100% | 1 318.2 / 976.1 / 887.3 | 5 260.8 / 3 575.1 / 1 797.1 |

### 8.1 三个结论

1. **所有 QoS、所有丢包率下到达率均 100%**——MQTT 跑在 TCP 之上，丢包由 TCP 重传兜底。对照理论：若无传输层重传，QoS 0 端到端到达率（双跳）应为 (1−p)²，15% 丢包时仅 72%。这纠正了"QoS 0 丢包即丢消息"的常见误解。
2. **延迟随丢包率显著上升**：0% 时亚毫秒级，15% 时约 1 秒——这是 TCP 重传等待的代价，也是不可靠网络下实时性下降的真实来源。
3. **高丢包下 QoS 2 延迟反而最低**（15%：QoS2 887 ms < QoS1 976 ms < QoS0 1 318 ms）：QoS 2 的应用层四次握手反压了发送速率，避免 TCP 发送缓冲区堆积导致的队头阻塞；QoS 0/1 猛发使缓冲区积压，单点丢包阻塞后续所有消息。

**QoS 的真正价值在连接中断场景**：持续连接下 TCP 已保证可靠，QoS 的差异体现在断连/重连时——见 §7，QoS 1 断线期 6 条消息排队补发、零丢失，QoS 0 则会在断连期间真正丢失。
