# 需求分析与设计指标

## 1. 编写目的

本项目选题为“基于 MQTT 的多节点环境监测系统设计”。在正式实现 MQTT Broker、软件模拟节点、Web 监控端以及后续 ESP32 硬件节点之前，需要先明确系统的应用场景、功能需求、性能指标和工程约束。

本文件作为课程设计报告中“需求分析与设计指标”章节的初稿，用于支撑后续总体方案设计、MQTT 主题结构设计、消息数据格式设计、系统测试方案和工程分析。

## 2. 应用场景

环境数据具有连续变化、节点分散、数据量单条较小但上传频繁等特点，适合使用 MQTT 这类轻量级发布/订阅协议进行传输。本项目的多节点环境监测系统可面向以下场景：

1. 校园/实验室环境监测：在教室、实验室、机房等区域布置节点，采集温度、湿度、光照等数据，辅助判断环境是否适合学习、实验或设备运行。
2. 智慧农业大棚：在农业大棚中监测温湿度和光照强度，为通风、灌溉、补光等控制策略提供数据依据。
3. 智能家居环境调节：在室内采集温度、湿度和光照数据，为空调、加湿器、窗帘、照明等设备联动提供基础信息。
4. 噪声检测与环境评价：在宿舍区、教学楼、道路附近等位置采集噪声数据，用于判断环境噪声是否超过设定范围。
5. 出行参考或公共空间环境提示：在公共区域采集温湿度、光照、噪声等数据，为用户提供当前环境状态提示，辅助出行或活动安排。

## 3. 系统目标

本系统的总体目标是设计并实现一个可验证、可扩展的 MQTT 多节点环境监测系统。系统初期使用 PC 软件模拟环境监测节点，后续逐步接入 ESP32 硬件节点采集真实环境数据。

系统需要达到以下目标：

1. 支持至少 2 个环境监测节点同时上传数据。
2. 支持温度、湿度、光照和噪声等环境数据的采集或模拟。
3. 使用 MQTT 协议完成节点到 Broker、Broker 到监控端的数据传输。
4. Web 监控端能够订阅 MQTT 主题并实时显示各节点数据。
5. 能够为后续 QoS 对比、消息延迟统计、消息到达率统计和节点断线重连测试提供基础。

## 4. 数据采集对象

本项目初步选取 4 类环境数据作为采集或模拟对象：

| 数据类型 | 英文字段建议 | 单位 | 数据来源 | 说明 |
| --- | --- | --- | --- | --- |
| 温度 | `temperature` | `C` | PC 模拟或温度传感器 | 反映环境冷热情况 |
| 湿度 | `humidity` | `%RH` | PC 模拟或湿度传感器 | 反映空气湿润程度 |
| 光照 | `light` | `lux` | PC 模拟或光照传感器 | 反映环境光照强度 |
| 噪声 | `noise` | `dB` | PC 模拟或噪声传感器 | 反映环境声音强度 |

每条环境数据应至少包含节点编号、数据类型、数据值、单位、上传时间和消息序号，便于监控端展示和后续测试统计。

## 5. 功能需求

### 5.1 多节点环境数据采集或模拟

系统至少包含 2 个环境监测节点。第一阶段可使用 PC 程序模拟节点，周期性生成温度、湿度、光照或噪声数据；后续可替换为 ESP32 节点采集真实传感器数据。

### 5.2 MQTT 数据发布

每个节点需要按照统一的 MQTT 主题结构发布环境数据。发布内容应采用统一消息格式，避免软件节点、硬件节点和监控端之间出现字段不一致的问题。

### 5.3 MQTT Broker 接收与转发

系统需要搭建本地或云端 MQTT Broker。Broker 负责接收各节点发布的数据，并将消息转发给订阅相关主题的监控端。

### 5.4 Web 监控端订阅与显示

Web 监控端需要订阅环境监测相关主题，并显示各节点的最新数据。显示内容至少包括节点编号、数据类型、数据值、单位和上传时间。

### 5.5 上传周期设置

节点应支持设置数据上传周期。不同上传周期下系统的运行情况可用于后续分析实时性、网络负载和稳定性。

### 5.6 节点异常与断线情况分析

系统需要考虑节点断开、网络恢复、重复消息、消息延迟等情况。后续测试中应记录节点断线后能否恢复上传，以及监控端是否能观察到异常状态或数据中断。

### 5.7 测试证据记录

系统运行过程中需要保留可验证证据，包括 MQTTX 测试记录、Wireshark 抓包、服务器或客户端日志、运行截图和测试数据表。

## 6. 性能指标

本项目第一版性能指标采用“可测试、可统计、可解释”的原则制定。具体目标值可在后续实际测试后调整。

| 指标 | 初步目标 | 测试方法 | 说明 |
| --- | --- | --- | --- |
| 消息到达率 | 不低于 95% | 统计节点发送消息数和监控端接收消息数，计算 `接收数 / 发送数 * 100%` | 用于评价 MQTT 消息传输可靠性 |
| 平均传输延迟 | 控制在 1 秒以内 | 消息中携带发送时间戳，监控端记录接收时间，计算多条消息平均延迟 | 用于评价系统实时性 |
| 断线重连恢复时间 | 控制在 10 秒以内 | 人为断开节点网络或 Broker 连接，恢复后记录节点重新成功发布数据所需时间 | 用于评价异常恢复能力 |
| 连续运行稳定性 | 连续运行 30 分钟无明显异常 | 多节点持续上传，记录断连、消息丢失、重复消息和程序异常情况 | 用于评价系统稳定运行能力 |
| 多节点并发能力 | 至少 2 个节点同时上传 | 同时启动多个节点，观察 Broker 和监控端是否能区分并显示各节点数据 | 用于评价系统扩展基础 |

## 7. 工程约束

### 7.1 实时性

环境监测数据需要及时上传和显示。上传周期过长会降低实时性，上传周期过短会增加网络负载和设备功耗，因此需要在实时性和系统负载之间进行权衡。

### 7.2 可靠性

系统需要考虑网络波动、节点掉线、Broker 暂时不可用等情况。MQTT 的 QoS 机制、重连机制和消息序号可用于辅助提高可靠性和分析异常。

### 7.3 准确性

软件模拟节点的数据应保持在合理范围内；硬件节点接入后，传感器误差、采样周期和环境干扰都会影响数据准确性，需要在测试中记录和分析。

### 7.4 可扩展性

系统后续可能增加更多节点和更多数据类型，因此 MQTT 主题结构、节点编号规则和消息格式应具有扩展能力，避免新增节点时大幅修改监控端逻辑。

### 7.5 功耗

ESP32 硬件节点接入后，Wi-Fi 通信、传感器采集频率和上传周期都会影响功耗。课程设计初期主要实现功能，但报告中需要分析功耗约束。

### 7.6 维护性

软件模拟节点、硬件节点和监控端应使用统一消息格式，并在文档中说明主题规则、字段含义和测试方法，便于小组成员协作维护。

## 8. 验证方式

| 需求或指标 | 验证方式 | 证据形式 |
| --- | --- | --- |
| 多节点同时上传 | 同时启动至少 2 个节点，观察监控端是否显示不同节点数据 | 运行截图、监控端日志 |
| MQTT 发布/订阅通信 | 使用 MQTTX 订阅主题并观察节点消息 | MQTTX 记录或截图 |
| Web 监控端显示 | 检查页面是否显示节点编号、数据类型、数据值、单位和上传时间 | 页面截图 |
| 消息到达率 | 对比节点发送日志和监控端接收日志 | 测试数据表、日志 |
| 平均传输延迟 | 统计消息发送时间和接收时间差值 | 测试数据表 |
| 断线重连恢复 | 人为断开节点网络或 Broker，恢复后记录重新上传时间 | 运行日志、测试记录 |
| QoS 效果分析 | 分别使用 QoS 0、QoS 1、QoS 2 发送消息并比较结果 | MQTTX 记录、Wireshark 抓包 |
| 协议过程分析 | 使用 Wireshark 抓取 MQTT 报文 | `.pcap` 抓包文件、分析截图 |

## 9. 小结

本阶段明确了多节点环境监测系统的应用场景、系统目标、数据采集对象、功能需求、性能指标和工程约束。后续设计应围绕这些需求展开，优先完成 MQTT 主题结构、消息数据格式、本地 Broker、软件模拟节点和 Web 监控端原型，再逐步开展 QoS、延迟、到达率、断线重连和 ESP32 硬件接入测试。
