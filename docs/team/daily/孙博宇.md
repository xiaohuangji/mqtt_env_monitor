## 今日遇到的问题

1. **ESP32 开发板型号与原引脚规划不一致**

   在前期设计中，曾参考普通 ESP32 开发板的引脚表，将 MQ-2、雨滴传感器和土壤湿度传感器分别规划到 GPIO32、GPIO33 和 GPIO35。但实际使用的开发板为 ESP-C3-32S，该型号并没有这些引脚，若继续使用原规划会导致程序与硬件连接不匹配。

2. **传感器模块接口选择存在取舍**

   MQ-2、MH-RD 和 FC-28 模块通常同时具有 AO 和 DO 接口。AO 输出模拟量，能够反映传感器检测值的连续变化；DO 输出数字量，通常只表示是否超过模块电位器设定的阈值。如果同时接入 AO 和 DO，会增加引脚占用和调试复杂度。

3. **MQTT JSON 字段格式需要严格统一**

   在调试过程中，为了观察传感器状态，曾考虑加入 `status`、`analog_value`、`digital_value` 等辅助字段。但根据前期《MQTT 主题与消息格式设计》，正式 JSON Payload 应统一使用 `node_id`、`data_type`、`value`、`unit`、`timestamp` 和 `seq` 这些字段。如果随意增加字段，可能会影响 Web 监控端后续解析。

4. **PlatformIO 中 ESP32-C3 串口配置问题**

   在编译和调试过程中，出现过 `Serial` 未定义或串口监视器只显示启动日志、不显示程序输出的问题。这与 ESP32-C3 的 USB CDC 串口配置有关。

5. **烧录时开发板未进入下载模式**

   在上传程序时，出现过开发板未进入下载模式的情况，导致程序无法正常烧录。这类问题通常与 ESP32-C3 的 BOOT 状态、复位时机以及外接传感器对启动电平的影响有关。

## 解决方案

1. **重新规划 ESP-C3-32S 引脚**

   根据 ESP-C3-32S 的实际引脚资源重新进行接口分配，将 MQ-2 气敏传感器 AO 接 GPIO0，MH-RD 雨滴传感器 AO 接 GPIO1，FC-28 土壤湿度传感器 AO 接 GPIO2，并同步修改 `config.h` 中的引脚定义，保证软件配置与硬件连接一致。

2. **优先采用 AO 模拟量接口采集**

   本次任务以基础数据采集和串口检测为主，因此优先接入 AO 模拟量接口。DO 数字量接口暂不接入，后续如果需要阈值报警功能，再单独扩展数字量检测逻辑。

3. **严格控制 JSON Payload 字段**

   正式 Payload 严格保留协议规定字段，只输出 `node_id`、`data_type`、`value`、`unit`、`timestamp` 和 `seq`。调试状态信息只作为串口辅助输出，不写入 JSON 消息中，保证数据格式与项目协议保持一致。

4. **调整 PlatformIO 工程配置**

   在 `platformio.ini` 中根据 ESP-C3-32S 的特点调整开发板配置，并加入相关 USB 串口编译选项，使工程能够正常编译。同时，通过观察串口启动日志判断开发板是否已经正常启动，再进一步排查程序串口输出通道问题。

5. **采用手动下载模式进行烧录**

   在程序上传时，通过按住 BOOT 键并配合 RST/EN 键的方式，使 ESP-C3-32S 进入下载模式后再进行烧录。为避免外接传感器影响启动电平，烧录时优先断开传感器信号线，烧录完成后再重新接入。
