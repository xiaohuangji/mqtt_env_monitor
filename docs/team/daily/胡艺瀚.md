3. 遇到的问题

今天主要遇到的问题包括：

- 系统总体方案图最初表达不够清晰，部分文字和箭头容易互相遮挡，且图中对节点采集数据类型的表达容易让人误解为不同节点只能采集固定数据。
- 在 GitHub 推送时，本地 `main` 分支和远程 `main` 分支存在差异，直接 `git push origin main` 被拒绝。
- 队友的 issue 和 PR 中存在任务范围偏大、文档内容重复、文件名缺少 `.md` 后缀、PR 与 issue 未关联等问题。
- 在 MQTTX CLI 验证时，PowerShell 直接传递 JSON 字符串容易导致双引号被处理掉，使 Payload 不再是标准 JSON。
- 使用 PowerShell 写入 JSON 文件时，如果使用默认 UTF-8 写法，可能会带有 BOM 字符，导致 MQTT Payload 前出现额外字符。

4. 解决方法

针对系统总体方案图的问题，重新调整了图和文字说明，明确 PC 模拟节点可以模拟多类环境数据，ESP32 硬件节点根据实际传感器能力采集一种或多种真实数据，避免将节点设计误解为固定单一类型采集。

针对 GitHub 推送被拒绝的问题，先执行 `git fetch origin` 获取远程更新，再使用 `git pull --ff-only origin main` 将本地分支快进到远程最新状态，之后再继续进行提交和推送。通过这个过程进一步理解了远程分支比本地分支更新时不能直接推送的原因。

针对队友 issue 和 PR 的问题，采用先审查、再评论建议、最后由对应负责人修改的方式处理。这样既能保持项目规范，又能保留每个成员自己的工作过程。

针对 MQTTX CLI 的 JSON 参数问题，最终采用 `--file-read` 从 JSON 文件读取 Payload 的方式，避免命令行直接传参导致引号丢失。针对 UTF-8 BOM 问题，记录了应使用 UTF-8 无 BOM 编码保存 JSON 文件的注意事项。
