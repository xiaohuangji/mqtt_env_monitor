原方案中包含雨量检测模块，但当前已有 DHT11 温湿度传感器，因此今天将 DHT11 温湿度传感器加入当前硬件节点设计中，并删除雨量检测模块相关内容。同时，根据新的传感器接入规划，对 GPIO0 至 GPIO4 的功能进行了重新分配，使当前文档中的硬件连接关系更加符合实际器材情况和后续程序开发需求。

此外，还对 WiFi 配网网页门户方案、传感器串口输出和 WiFi 连接过程相关文档进行了整理，为后续 ESP32-C3 节点接入 MQTT Broker、实现无线数据上传和长期数据记录提供文档基础。

本人完成具体任务
今天首先修改了 ESP32-C3-32S 硬件节点传感器接入文件中的串口连接和引脚分配部分。修改后的引脚规划为：GPIO0 连接 DHT11 温湿度传感器，GPIO1 预留给后续光照传感器，GPIO2 预留给后续噪声检测模块，GPIO3 连接土壤湿度检测模块，GPIO4 连接 MQ-2 气体传感器模块。该分配方案既满足当前已有传感器的接入需求，也为后续扩展光照和噪声检测功能保留了接口。

其次，对各个模块进行了相应介绍。DHT11 温湿度传感器用于采集环境温度和空气湿度，可输出 temperature 和 humidity 两类数据；土壤湿度检测模块用于检测土壤干湿程度，对应 soil_moisture 数据类型；MQ-2 气体传感器模块用于检测烟雾、可燃气体等气体浓度变化，对应 gas 数据类型。光照传感器和噪声检测模块当前仅作为预留模块，在文档中说明其后续可分别用于 light 和 noise 数据采集。

第三，根据当前已有 DHT11 温湿度传感器的情况，将 DHT11 模块加入程序设计说明中，并删除雨量检测模块相关内容。调整后，当前硬件节点的主要采集对象由原先的气体、雨量、土壤湿度，修改为温度、湿度、土壤湿度和气体检测数据。由于 DHT11 一个模块可以同时采集温度和湿度，因此虽然硬件上接入的是三个主要传感器模块，但实际数据类型增加为四类。

最后，对 WiFi 配网网页门户设计说明和串口输出过程总结进行了梳理。明确 ESP32-C3 启动后可以读取已保存 WiFi 信息并自动连接热点，若没有保存信息或连接失败，则进入 AP 配网模式，通过网页输入 WiFi 名称和密码。串口输出方面，继续保持 Topic 和 JSON Payload 的设计思路，为后续 MQTT 发布提供统一的数据格式基础。

遇到的问题
问题1：原有硬件节点文档中的传感器接入内容与当前实际器材不完全一致。文档中原先包含雨量检测模块，但当前实际新增的是 DHT11 温湿度传感器，因此需要重新调整传感器接入说明。

问题2：ESP32-C3-32S 的 GPIO 引脚数量有限，需要在当前传感器和后续扩展模块之间进行合理分配。如果不提前规划，后续接入光照、噪声等传感器时容易出现引脚冲突。

问题3：DHT11 与土壤湿度模块、MQ-2 气体传感器的接口类型不同。DHT11 采用数字数据接口，而土壤湿度模块和 MQ-2 气体传感器主要使用 AO 模拟输出，因此在文档和程序说明中需要区分不同传感器的采集方式。

解决方法
解决1：针对传感器接入内容不一致的问题，重新修改 ESP32-C3-32S 硬件节点传感器接入文件，将雨量检测模块从当前接入方案中删除，并新增 DHT11 温湿度传感器模块。修改后，当前文档能够准确反映现有硬件器材和当前阶段的实际开发内容。

解决2：针对引脚分配问题，对 GPIO0 至 GPIO4 进行了统一规划。GPIO0 用于 DHT11 温湿度传感器，GPIO3 用于土壤湿度检测模块，GPIO4 用于 MQ-2 气体传感器模块；GPIO1 和 GPIO2 分别作为光照传感器和噪声检测模块的预留接口。这样既满足当前采集需求，也为后续功能扩展留下空间。

解决3：针对接口类型不同的问题，在文档中分别说明 DHT11 的 DATA 数字数据接口，以及土壤湿度模块、MQ-2 气体传感器 AO 模拟输出接口的作用。通过这种方式，使后续编写程序时能够清楚区分数字传感器读取和 ADC 模拟量采集的不同逻辑。
