6月12日工作日志
1. 今日主要完成工作内容

今日主要围绕 ESP32-C3 AP WiFi 配网网页门户测试过程中出现的问题进行分析与总结。本阶段采用 AP 配网网页门户方案，目的是避免在 ESP32-C3 程序中硬编码 WiFi 名称和密码，使设备能够通过网页方式完成 WiFi 配置。基本测试流程为：ESP32-C3 上电后开启临时 AP 热点，手机连接该热点后访问 http://192.168.4.1，进入配网页面并填写目标 WiFi 的名称和密码，提交后由 ESP32-C3 保存 WiFi 信息并尝试连接目标热点。

在实际测试过程中，配网流程并不稳定，主要表现为手机浏览器无法稳定打开配网页面、填写 Windows 移动热点名称和密码后页面丢失或无响应、无法直接判断 ESP32-C3 是否成功保存热点信息并重启等问题。由于 WiFi 配网是后续 MQTT 联调的前提，如果该问题不能解决，ESP32-C3 将无法稳定接入网络，也就无法继续进行 NTP 时间同步、MQTT Broker 连接和传感器数据上传。因此，今日重点对这些现象进行整理，并分析可能原因和对应解决方法。

2. 本人完成具体任务

本人首先对 ESP32-C3 AP 配网网页门户的测试流程进行了复现。将程序烧录到 ESP32-C3 后，通过串口监视器观察设备是否进入 AP 配网模式，并确认 ESP32-C3 能够创建用于配网的临时热点。随后使用手机连接 ESP32-C3 创建的热点，并尝试在浏览器中访问 http://192.168.4.1 配网页面。

在多次测试中发现，手机虽然能够连接到 ESP32-C3 的 AP 热点，但网页访问并不稳定。有时浏览器能够正常进入配网页面，有时页面会长时间加载或提示无法访问。进入页面后，填写 Windows 移动热点名称和密码并提交，网页也经常出现丢失、卡住或无响应的情况，导致无法从网页端直接判断 WiFi 信息是否已经提交成功。

针对上述情况，本人结合 ESP32-C3 的运行机制进行了分析。ESP32-C3 在接收到网页提交的 WiFi 信息后，可能会将 SSID 和密码保存到本地存储中，然后执行重启操作。设备重启后，原本由 ESP32-C3 创建的 AP 热点会短暂关闭，手机与 ESP32-C3 的连接也会断开，所以浏览器页面出现丢失并不一定代表配网失败。同时，当 ESP32-C3 从 AP 模式切换到 STA 模式后，原来的 192.168.4.1 地址不再可用，后续需要通过串口输出的新 IP 地址判断设备是否连接成功。

本人还分析了校园网覆盖环境可能对测试造成的影响。虽然手机访问 http://192.168.4.1 理论上只依赖 ESP32-C3 自建热点，不需要连接校园网，但在实际测试环境中，校园网信号覆盖较强，手机可能会因为 ESP32-C3 热点无法访问互联网而自动切换到校园网、移动数据或其他已保存 WiFi。这样会导致浏览器请求不再发送给 ESP32-C3，从而出现网页丢失、无响应或无法访问的情况。

3. 遇到的问题

第一，手机浏览器访问 http://192.168.4.1 时网页无法稳定打开。手机虽然已经连接 ESP32-C3 创建的 AP 热点，但浏览器页面有时长时间加载，有时直接提示无法访问。

第二，填写 Windows 移动热点名称和密码并提交后，网页可能会直接丢失、卡住或无响应，导致无法从网页端确认 ESP32-C3 是否保存 WiFi 信息并重启。

第三，校园网覆盖环境可能对配网过程造成间接影响。由于 ESP32-C3 的 AP 热点通常不能访问互联网，手机可能自动切换到校园网、移动数据或其他已保存网络，导致配网页面访问中断。

4. 解决方法

第一，针对网页无法稳定打开的问题，测试时需要确认手机已经真正连接到 ESP32-C3 配网热点，并在浏览器中明确输入 http://192.168.4.1，避免浏览器自动跳转到 HTTPS。同时，当手机提示该 WiFi 无法访问互联网时，应选择“仍然连接”或“保持连接”。

第二，针对提交后页面丢失的问题，不能只依靠网页端反馈判断配网结果，而应结合串口监视器观察 ESP32-C3 的实际状态。重点查看串口中是否出现 WiFi 信息保存、设备重启、读取已保存 WiFi、尝试连接目标热点以及连接成功等提示信息。

第三，针对校园网覆盖环境影响的问题，测试时应关闭手机移动数据，避免手机自动连接校园网或其他已保存 WiFi，保证手机稳定连接 ESP32-C3 的 AP 配网热点。必要时可以使用电脑连接 ESP32-C3 热点进行配网测试，减少手机自动切换网络造成的影响。

5. 今日成果

今日完成了 ESP32-C3 AP WiFi 配网网页门户测试问题的复现、分析和总结。通过多次测试，明确了配网页面访问不稳定、提交后页面丢失、网页无响应和手机自动切换网络等主要问题。经过分析，认为这些问题并不完全是 ESP32-C3 程序错误，也可能与 ESP32-C3 保存配置后重启、AP 模式切换到 STA 模式、手机网络自动切换机制以及校园网覆盖环境有关。

今日还明确了后续调试思路：网页主要用于提交 WiFi 信息，ESP32-C3 是否真正保存成功和连接成功，应以串口监视器输出为主要判断依据。同时，总结了测试时需要注意的操作方式，例如保持手机连接 ESP32-C3 热点、关闭移动数据、避免自动连接校园网以及使用 http://192.168.4.1 访问配网页面。

6. 后续任务规划

下一步将根据今日总结的问题继续对 AP 配网网页门户进行优化。重点改进方向包括：增加更清晰的串口状态提示，使配网流程中每一步状态更加明确；优化网页提交后的反馈，避免用户误认为页面丢失就是配网失败；同时考虑改进 WiFi 选择方式，减少用户手动输入 WiFi 名称时可能出现的错误。后续将继续验证 ESP32-C3 是否能够稳定保存 WiFi 信息，并在重启后自动连接目标热点。

6月13日工作日志
1. 今日主要完成工作内容

今日主要围绕 ESP32-C3 AP WiFi 配网网页门户的优化工作展开。在前一天完成问题分析与总结的基础上，今日重点对配网方式进行改进，将原先需要用户手动输入 WiFi 名称的方式，优化为自动扫描附近 WiFi 并在网页中显示可选热点列表。用户进入配网页面后，可以直接选择目标 WiFi，再输入对应密码完成配置。

该优化的目的在于减少手动输入 WiFi 名称带来的错误，提高配网成功率。原先的配网方式需要用户手动填写 WiFi SSID，如果热点名称中包含空格、大小写差异或特殊字符，容易造成输入不一致，从而导致 ESP32-C3 连接失败。通过自动扫描附近 WiFi 并提供选择列表，可以让 ESP32-C3 直接获取准确的 SSID，用户只需要选择目标热点并输入密码即可，配网流程更加直观和可靠。

2. 本人完成具体任务

本人首先根据前一天总结的问题，对配网网页门户的操作方式进行了优化分析。原有方案中，用户需要在网页中手动填写 Windows 移动热点名称和密码。由于 SSID 输入容易出现大小写错误、空格遗漏或特殊字符不一致等问题，因此将优化方向确定为：ESP32-C3 在进入 AP 配网模式后，主动扫描附近可用 WiFi，并将扫描到的 WiFi 名称显示在网页中，供用户直接选择。

在优化思路中，ESP32-C3 首先开启 AP 配网热点，手机连接该热点后访问 http://192.168.4.1。进入网页后，网页不再只提供手动输入框，而是显示 ESP32-C3 扫描到的附近 WiFi 列表。用户可以从列表中选择目标 WiFi，例如 Windows 移动热点，然后输入对应密码并提交。ESP32-C3 接收到选择结果后，将目标 WiFi 的 SSID 和密码保存到本地存储中，并重启或切换到 STA 模式尝试连接目标网络。

在测试过程中，本人继续按照“网页提交 + 串口确认”的方式判断配网结果。提交 WiFi 信息后，即使网页短暂丢失或无响应，也不再直接判断为失败，而是通过串口监视器观察 ESP32-C3 是否保存配置、是否重启、是否读取到已保存的热点信息以及是否开始连接目标 WiFi。串口输出显示 ESP32-C3 能够尝试连接已保存 WiFi，并在连接成功后输出 IP 地址，说明配网流程最终完成。

今日还继续针对校园网覆盖环境进行了测试注意事项整理。由于 ESP32-C3 的 AP 热点无法访问互联网，手机仍可能出现自动切换网络的情况。因此在优化后的测试过程中，仍需要关闭手机移动数据，并在手机提示该 WiFi 无法访问互联网时选择保持连接。通过减少手机自动切网和减少 SSID 手动输入错误两个方面共同优化，提高了配网过程的成功率。

3. 遇到的问题

第一，原先手动输入 WiFi 名称的方式容易出现 SSID 输入错误，例如大小写不一致、空格遗漏或特殊字符输入错误，导致 ESP32-C3 无法连接目标热点。

第二，网页提交后仍可能出现短暂丢失或无响应现象，用户如果只看网页端反馈，容易误认为配网失败。

第三，校园网覆盖环境下，手机仍可能因为 ESP32-C3 热点无法访问互联网而自动切换到校园网或移动数据，影响配网页面的稳定访问。

4. 解决方法

第一，将配网方式优化为自动扫描附近 WiFi 并显示可选列表。用户不再需要手动输入 WiFi 名称，而是直接选择扫描到的目标热点，只需要输入密码即可完成配置，从而减少 SSID 输入错误。

第二，继续采用“网页提交 + 串口确认”的方式判断配网是否成功。网页用于提交 WiFi 信息，ESP32-C3 是否保存成功、是否重启、是否连接目标 WiFi，主要通过串口监视器中的日志进行确认。

第三，测试时保持手机稳定连接 ESP32-C3 的 AP 热点，关闭手机移动数据，避免自动连接校园网或其他已保存 WiFi。当手机提示该网络无法访问互联网时，选择继续连接或保持连接，以减少校园网覆盖环境对配网页面的影响。

5. 今日成果

今日完成了 ESP32-C3 AP WiFi 配网网页门户的优化方案整理，将原先手动输入 WiFi 名称的方式改进为自动扫描附近 WiFi 并选择连接的方式。该优化减少了用户输入 SSID 的步骤，降低了因热点名称输入错误导致连接失败的概率，提高了配网流程的便利性和稳定性。

经过多次调试后，ESP32-C3 最终能够通过配网页面获取目标 WiFi 信息，保存配置后读取已保存热点，并成功连接目标网络。串口中能够输出连接成功信息和设备 IP 地址，说明 ESP32-C3 已经完成 WiFi 接入，AP 配网网页门户方案基本可行。

通过本次优化，进一步明确了后续 AP 配网设计思路：网页端应尽量减少用户手动输入内容，串口端应提供清晰的状态输出，网络测试时应尽量避免手机自动切换网络。该结果为后续 NTP 时间同步、MQTT Broker 连接和传感器数据上传提供了网络基础。
