# 专业知识汇总

## MQTT 基础

MQTT 是一种轻量级的发布/订阅式消息传输协议，常用于物联网场景。系统中的设备不直接互相通信，而是通过 MQTT Broker 转发消息。

主要角色：

- Publisher：发布者，负责向某个主题发布消息，例如环境监测节点。
- Subscriber：订阅者，负责订阅主题并接收消息，例如监控端。
- Broker：消息代理服务器，负责接收、转发和管理消息。
- Topic：主题，用于组织消息路径。

## 发布/订阅模型

在本项目中，环境监测节点作为发布者，监控端作为订阅者。节点将温度、湿度、光照、噪声等数据发布到指定主题，监控端订阅对应主题后接收并展示数据。

示例主题：

```text
classroom/node01/temperature
classroom/node02/humidity
classroom/+/temperature
classroom/# 
```

其中 `+` 表示单层通配符，`#` 表示多层通配符。

## MQTT 主题设计原则

- 层次清晰：建议按场景、节点、数据类型分层。
- 便于扩展：后续增加节点或数据类型时不需要修改整体结构。
- 便于订阅：监控端可以订阅单个节点、单类数据或全部数据。
- 命名统一：使用小写英文、数字和分隔符，避免中英文混用。

建议初版主题：

```text
env_monitor/node01/temperature
env_monitor/node01/humidity
env_monitor/node02/light
env_monitor/node02/noise
env_monitor/+/+
```

## QoS 机制

MQTT 支持 3 种 QoS 等级：

| QoS | 含义 | 特点 | 适用场景 |
| --- | --- | --- | --- |
| QoS 0 | 最多一次 | 开销小，不保证到达 | 高频、可容忍少量丢失的传感器数据 |
| QoS 1 | 至少一次 | 保证到达，可能重复 | 重要环境数据、告警数据 |
| QoS 2 | 只有一次 | 最可靠，开销最大 | 对重复极其敏感的业务 |

本项目可将 QoS 0 作为普通数据默认等级，将 QoS 1 用于告警或关键数据测试，QoS 2 用于对比分析。

## MQTT 与 HTTP 对比

| 维度 | MQTT | HTTP |
| --- | --- | --- |
| 通信模型 | 发布/订阅 | 请求/响应 |
| 连接方式 | 长连接 | 通常为短请求或复用连接 |
| 多节点扩展 | 适合多节点同时上传 | 节点多时服务端接口压力更明显 |
| 消息开销 | 较小 | 报文头较大 |
| 实时性 | Broker 推送，实时性较好 | 客户端请求或轮询 |
| 典型场景 | 物联网、传感器、低带宽网络 | Web API、业务系统接口 |

对于本项目的多节点环境监测场景，MQTT 更适合处理多个节点持续上传和监控端实时订阅。

## 节点掉线与重连

需要考虑的问题：

- 节点 Wi-Fi 断开后如何恢复连接。
- MQTT Broker 暂时不可用时如何重连。
- 节点掉线后监控端如何显示状态。
- 重连后是否继续按原主题发布数据。

可选机制：

- 使用 MQTT keep alive 检测连接状态。
- 使用 Last Will 消息发布节点离线状态。
- 节点端增加自动重连逻辑。
- 数据包中增加时间戳和序号，便于发现延迟、重复和丢失。

## 消息格式建议

初版消息载荷可使用 JSON：

```json
{
  "node_id": "node01",
  "data_type": "temperature",
  "value": 26.5,
  "unit": "C",
  "timestamp": "2026-06-08T15:00:00+08:00",
  "seq": 1
}
```

字段说明：

- `node_id`：节点编号。
- `data_type`：数据类型。
- `value`：数据值。
- `unit`：单位。
- `timestamp`：节点上传时间。
- `seq`：消息序号，用于统计丢失和重复。

## ESP32 环境数据采集

后续硬件节点可使用 ESP32 连接环境传感器采集真实数据。

可选传感器：

- DHT11 或 DHT22：温湿度。
- BH1750：光照强度。
- 模拟声音传感器：噪声强度。

ESP32 需要完成：

- 连接 Wi-Fi。
- 读取传感器数据。
- 按设定周期发布 MQTT 消息。
- 在网络断开时自动重连。
- 通过串口输出调试日志。

## MQTTX 测试方法

MQTTX 可用于连接 Broker、订阅主题、发布测试消息和观察消息内容。

常用操作：

1. 新建连接，填写 Broker 地址、端口、Client ID。
2. 订阅 `env_monitor/+/+` 查看所有节点数据。
3. 手动发布测试 JSON 消息。
4. 切换 QoS 等级，观察消息接收情况。
5. 保存测试截图或记录到测试证据目录。

## Wireshark 抓包要点

Wireshark 可用于分析 MQTT 通信过程。

常用过滤条件：

```text
mqtt
tcp.port == 1883
```

建议关注：

- CONNECT、CONNACK：客户端连接过程。
- PUBLISH：节点发布消息。
- SUBSCRIBE、SUBACK：监控端订阅过程。
- PINGREQ、PINGRESP：心跳保活。
- QoS 1 下的 PUBACK。
- QoS 2 下的 PUBREC、PUBREL、PUBCOMP。
