# ESP32 硬件节点接线与采集流程

## 1. 编写目的

本项目选题为“基于 MQTT 的多节点环境监测系统设计”。在 PC 模拟节点和 Web 监控端实现之前，需要先明确 ESP32 硬件节点的组成、传感器选型、引脚接线方式以及数据采集发布流程。

本文档仅聚焦 ESP32 硬件节点设计，不重复展开 MQTT Topic 命名规则、JSON 字段规范、节点编号规则和数据类型命名规则。相关通信格式统一参考《MQTT 主题与消息格式设计》。

## 2. 硬件节点基本组成

第一版 ESP32 硬件节点由以下硬件组成：

| 硬件模块 | 作用 |
| --- | --- |
| ESP32 开发板 | 核心控制器，负责 Wi-Fi 连接、传感器数据采集、JSON 封装和 MQTT 发布 |
| 温湿度传感器 | 采集环境温度和空气相对湿度 |
| 光照传感器 | 采集环境光照强度 |
| 噪声传感器 | 采集环境噪声强度 |
| 面包板和杜邦线 | 连接 ESP32 与各类传感器 |
| USB 数据线 | 用于 ESP32 供电、程序下载和串口调试 |
| MQTT Broker | 接收 ESP32 发布的数据，并转发给 Web 监控端 |

ESP32 硬件节点的主要任务是采集真实环境数据，并按照项目统一规定的 MQTT Topic 和 JSON 消息格式发布数据。

## 3. 第一版传感器选型

第一版 ESP32 硬件节点计划采集温度、湿度、光照和噪声四类环境数据。

| 采集内容 | 推荐传感器 | 对应 `data_type` | 对应 `unit` | 说明 |
| --- | --- | --- | --- | --- |
| 温度 | DHT11 / DHT22 | `temperature` | `C` | 表示环境温度 |
| 湿度 | DHT11 / DHT22 | `humidity` | `%RH` | 表示空气相对湿度 |
| 光照 | BH1750 | `light` | `lux` | 表示环境光照强度 |
| 噪声 | 分贝检测模块 / 噪声传感器模块 | `noise` | `dB` | 表示环境噪声强度 |

说明：

- ESP32 硬件节点中的 `data_type` 和 `unit` 必须与《MQTT 主题与消息格式设计》保持一致。
- 如果第一版使用普通模拟声音传感器，实际读取值可能是 ADC 原始值。为了符合项目通信规范，建议后续采用支持分贝输出或可换算为分贝值的噪声模块。

## 4. ESP32 与传感器接线方案

### 4.1 DHT11 / DHT22 温湿度传感器接线

| 传感器引脚 | ESP32 引脚 | 说明 |
| --- | --- | --- |
| VCC | 3V3 | 传感器供电 |
| GND | GND | 接地 |
| DATA | GPIO 4 | 温湿度数据引脚 |

说明：

- DHT11 / DHT22 使用单总线方式与 ESP32 通信。
- DHT 传感器一次读取可以得到温度和湿度两个值。
- 发布 MQTT 消息时，温度和湿度应拆分为两条独立消息。

### 4.2 BH1750 光照传感器接线

| 传感器引脚 | ESP32 引脚 | 说明 |
| --- | --- | --- |
| VCC | 3V3 | 传感器供电 |
| GND | GND | 接地 |
| SDA | GPIO 21 | I2C 数据线 |
| SCL | GPIO 22 | I2C 时钟线 |

说明：

- BH1750 使用 I2C 通信方式。
- ESP32 常用 I2C 引脚为 GPIO 21 和 GPIO 22。
- BH1750 可直接读取光照强度，适合作为第一版光照采集传感器。

### 4.3 噪声传感器接线

| 传感器引脚 | ESP32 引脚 | 说明 |
| --- | --- | --- |
| VCC | 3V3 | 传感器供电 |
| GND | GND | 接地 |
| AO / OUT | GPIO 35 | 噪声数据输入引脚 |

说明：

- ESP32 可通过 ADC 引脚读取噪声传感器输出。
- 第一版建议选择能够输出噪声强度或便于换算为分贝值的模块。
- 如果暂时只能读取模拟原始值，应在程序或测试说明中注明其为近似值。

## 5. ESP32 节点编号建议

根据项目已有编号规则，PC 模拟节点已建议使用：

| 节点编号 | 节点类型 | 说明 |
| --- | --- | --- |
| `node01` | PC 模拟节点 | 用于模拟温度、湿度等数据 |
| `node02` | PC 模拟节点 | 用于模拟光照、噪声等数据 |

因此，第一版 ESP32 硬件节点建议使用：

| 节点编号 | 节点类型 | 说明 |
| --- | --- | --- |
| `node03` | ESP32 硬件节点 | 用于采集真实温度、湿度、光照和噪声数据 |

后续新增 ESP32 节点时，可继续使用 `node04`、`node05` 等编号。

## 6. ESP32 发布数据对应关系

若 ESP32 硬件节点编号为 `node03`，则第一版采集数据与发布 Topic 的对应关系如下：

| 传感器 | 采集值 | `data_type` | `unit` | 发布 Topic |
| --- | --- | --- | --- | --- |
| DHT11 / DHT22 | 温度 | `temperature` | `C` | `env_monitor/node03/temperature` |
| DHT11 / DHT22 | 湿度 | `humidity` | `%RH` | `env_monitor/node03/humidity` |
| BH1750 | 光照强度 | `light` | `lux` | `env_monitor/node03/light` |
| 噪声传感器 | 噪声强度 | `noise` | `dB` | `env_monitor/node03/noise` |

说明：

- 每条 MQTT 消息只表示一种数据类型。
- DHT11 / DHT22 虽然一次读取两个值，但发布时应拆分为温度消息和湿度消息。
- Topic 和 Payload 字段格式不在本文档中重复定义，统一参考《MQTT 主题与消息格式设计》。

## 7. ESP32 数据采集与发布流程

ESP32 硬件节点从采集数据到发布 MQTT 消息的流程如下：

```text
ESP32 上电
  ↓
初始化串口
  ↓
初始化 Wi-Fi 配置
  ↓
初始化 MQTT 客户端配置
  ↓
初始化 DHT11 / DHT22 温湿度传感器
  ↓
初始化 BH1750 光照传感器
  ↓
初始化噪声传感器输入引脚
  ↓
连接 Wi-Fi
  ↓
连接 MQTT Broker
  ↓
周期性采集传感器数据
  ↓
读取温度、湿度、光照、噪声数据
  ↓
根据采集值确定 data_type、value、unit
  ↓
生成 timestamp 和 seq
  ↓
按照统一 JSON 基础字段封装 Payload
  ↓
按照统一 Topic 结构生成发布主题
  ↓
发布 MQTT 消息
  ↓
串口输出 Topic 和 Payload，便于调试
  ↓
等待下一次采集周期
```

建议第一版采集周期设置为 5 秒。ESP32 每隔 5 秒采集一次传感器数据，并分别发布温度、湿度、光照和噪声数据。

## 8. ESP32 串口调试输出建议

ESP32 在发布 MQTT 消息时，应通过串口输出当前发布的 Topic 和 Payload，便于检查数据格式是否符合项目统一规范。

示例输出如下：

```text
WiFi connected
MQTT connected

Publish topic: env_monitor/node03/temperature
Payload: {"node_id":"node03","data_type":"temperature","value":26.5,"unit":"C","timestamp":"2026-06-08T22:30:00+08:00","seq":1}

Publish topic: env_monitor/node03/humidity
Payload: {"node_id":"node03","data_type":"humidity","value":58.2,"unit":"%RH","timestamp":"2026-06-08T22:30:05+08:00","seq":2}

Publish topic: env_monitor/node03/light
Payload: {"node_id":"node03","data_type":"light","value":420,"unit":"lux","timestamp":"2026-06-08T22:30:10+08:00","seq":3}

Publish topic: env_monitor/node03/noise
Payload: {"node_id":"node03","data_type":"noise","value":48.6,"unit":"dB","timestamp":"2026-06-08T22:30:15+08:00","seq":4}
```

通过串口输出可以检查：

1. Topic 是否与硬件节点编号和数据类型一致。
2. `node_id` 是否与 Topic 中的节点编号一致。
3. `data_type` 是否与 Topic 最后一层一致。
4. `unit` 是否与项目规定的数据单位一致。
5. `timestamp` 是否正常生成。
6. `seq` 是否递增。

## 9. 后续传感器扩展方式

后续如果需要在 ESP32 节点上扩展更多传感器，应保持项目统一 Topic 结构和 JSON 基础字段不变，只新增数据类型和对应单位。

可扩展传感器示例：

| 扩展数据 | 推荐传感器 | 建议 `data_type` | 说明 |
| --- | --- | --- | --- |
| 气压 | BMP280 / BME280 | `pressure` | 表示环境气压 |
| 二氧化碳浓度 | CCS811 / SCD40 | `co2` | 表示空气中二氧化碳浓度 |
| PM2.5 | PMS5003 | `pm25` | 表示细颗粒物浓度 |

说明：

- 新增 `data_type` 和 `unit` 前，应先在《MQTT 主题与消息格式设计》中统一规定。
- ESP32 程序中应将不同传感器的采集逻辑封装为独立函数，方便后续扩展。
- Web 监控端应通过数据类型映射表显示中文名称。

## 10. 与后续模块的关系

ESP32 硬件节点设计需要与后续模块保持接口一致。

| 后续模块 | 与 ESP32 硬件节点的关系 |
| --- | --- |
| PC 模拟节点 | 与 ESP32 使用相同 Topic 和 JSON 格式，便于统一测试 |
| MQTT Broker | 接收 ESP32 发布的环境数据 |
| Web 监控端 | 订阅环境监测数据并解析 JSON 字段进行展示 |
| 测试方案 | 使用 `timestamp` 和 `seq` 统计延迟、到达率、重复和乱序情况 |

ESP32 硬件节点只需要按照统一格式发布数据，Web 监控端无需关心数据来自 PC 模拟节点还是真实 ESP32 硬件节点。

## 11. 小结

第一版 ESP32 硬件节点以 ESP32 开发板为核心，连接温湿度传感器、光照传感器和噪声传感器，实现真实环境数据采集。

本文档重点明确了：

- ESP32 硬件节点由哪些模块组成。
- 第一版使用哪些传感器。
- ESP32 与各传感器的接线方式。
- ESP32 节点建议编号。
- 采集数据与发布消息之间的对应关系。
- ESP32 从采集数据到发布 MQTT 消息的流程。
- 后续扩展更多传感器的方式。

MQTT Topic、JSON 字段、节点编号和数据类型命名的详细规则统一参考《MQTT 主题与消息格式设计》，本文档不再重复展开。
